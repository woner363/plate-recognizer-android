## 1. 结论摘要

v0.2.9 相比之前版本已经修复了多处关键风险：识别 UI 状态改为 Room Session 单一来源、保存链路引入 expected-state 状态迁移、`sourceSessionId` 唯一索引用于幂等保存、发布流程增加了签名证书与 `versionName` 校验，整体代码质量比 v0.2.7 明显提升。

本次没有发现“立刻阻断 APK 使用”的 P0 问题，但仍有几类值得尽快处理的问题：

| 优先级 | 问题 | 影响 |
|---|---|---|
| P1 | `FAILED` session 状态没有 UI 出口，且失败原因没有真正写入数据库 | 进程恢复后的失败任务可能被用户看不见、清不掉，相关图片也可能长期被孤儿清理保护 |
| P1 | 缺少 Room Migration / DAO 状态机 / 进程恢复链路的 Android instrumentation 测试 | 当前核心可靠性大量依赖代码推理，数据库迁移和事务行为没有真实 SQLite 回归保障 |
| P2 | `Discarding` 期间确认对话框仍可交互 | 用户可能在放弃过程中看到可点击保存/取消按钮，虽然 ViewModel 会 no-op，但体验与状态语义不一致 |
| P2 | `setRecognized()` 返回值被忽略 | 状态迁移失败时没有补偿或错误提示，极端并发/恢复场景下 UI 可能停在处理中 |
| P2 | 发布链路仍缺少 `versionCode` 单调递增校验、Action SHA 固定、生产签名策略 | GitHub Release 已能构建，但发布治理还不够硬 |
| P2 | `compileSdk/targetSdk` 与依赖版本偏旧，Lint 仍有 70 个 warning | 长期维护、适配新系统和上架渠道会受影响 |

## 4. 当前仍存在的问题

### P1-1：`FAILED` 状态没有真正的用户出口，且失败原因没有写入数据库

#### 现象

`PlatesViewModel` 在进程恢复时会把 `CAPTURING` / `RECOGNIZING` 的未完成 session 标记为 `FAILED`：

- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:146-149`

UI 映射也定义了 `RecognitionUiState.Failed`：

- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:408-409`

但 `MainScreen` 只把下面三种状态纳入 `pendingSession`：

- `AwaitingConfirmation`
- `Saving`
- `Discarding`

位置：

- `app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt:41-45`

也就是说，`FAILED` 状态不会展示确认对话框，也没有“放弃失败任务 / 删除图片 / 重试识别”的明确按钮。虽然 `launchRecognition()` 允许在 `Failed` 状态下开始新识别：

- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:255-263`

但这属于隐式清理：用户看不到当前有一个失败任务，也不知道旧图片会被怎样处理。

更关键的是，`RecognitionSessionRepository.markFailed(id, error)` 接收了 `error` 参数，但实际使用的是只更新 `state` / `updated_at` 的 `transitionIf()`：

- `app/src/main/java/com/example/platerecognizer/data/RecognitionSessionRepository.kt:102-113`
- `app/src/main/java/com/example/platerecognizer/data/PlateDao.kt:89-101`

因此传入的 `"上次识别未完成"` / `"任务被并发终止"` 并不会写入 `recognition_sessions.error`。UI 最终只能看到默认文案 `"识别任务失败"`。

#### 影响

- 进程恢复后的失败任务对用户不可见；
- `FAILED` 属于非终态，`listActiveImageUris()` 会继续把它的图片 URI 视为活跃引用：
  - `app/src/main/java/com/example/platerecognizer/data/PlateDao.kt:65-72`
- 如果用户不再发起新识别，失败 session 与对应图片可能长期保留；
- 排查失败原因时缺少具体错误信息。

#### 建议

1. 新增 DAO 方法，例如：

```kotlin
@Query("""
    UPDATE recognition_sessions
    SET state = 'FAILED',
        error = :error,
        updated_at = :updatedAt
    WHERE id = :id AND state = :expectedState
""")
suspend fun markFailedIf(
    id: String,
    expectedState: SessionState,
    error: String?,
    updatedAt: Long,
): Int
```

2. 让 `markFailed()` 使用这个方法，确保失败原因入库。

3. 在 `MainScreen` 对 `RecognitionUiState.Failed` 显示一个轻量错误面板：

- 显示失败原因；
- 提供“重新识别”；
- 提供“清除本次任务”或“删除失败图片”；
- 清除时将 session 标记为 `DISCARDING -> DISCARDED` 并删除图片。

4. 如果决定 `FAILED` 不需要持久保留，也可以把它改成终态，失败时直接删除 session / 图片；但这会丢失恢复提示，体验不如显式错误面板。

#### 验收标准

- 进程在识别中被杀后重启，用户能看到明确失败提示；
- 用户可以一键清理失败任务；
- `recognition_sessions.error` 中能看到具体失败原因；
- 失败 session 不会无限期保护图片文件。

### P2-1：`Discarding` 期间确认对话框仍然可交互

#### 现象

`MainScreen` 将 `Discarding` 也纳入 `pendingSession`：

- `app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt:41-45`

但禁用对话框只判断 `isSaving`：

- `app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt:45`
- `app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt:152`

这意味着进入 `Discarding` 后，确认对话框仍可能显示，且保存/取消按钮仍是可点击状态。ViewModel 端因为 `confirmPending()` / `discardPending()` 只接受 `AwaitingConfirmation`，最终大多会 no-op：

- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:289-291`
- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:327-329`

但 UI 呈现与实际状态不一致。

#### 影响

- 用户在点击“取消”后短时间内仍看到可交互对话框；
- 如果图片删除较慢，用户可能误以为保存还能生效；
- 这类“按钮可点但无反应”的体验会降低信任感。

#### 建议

把禁用条件从 `isSaving` 扩展为“正在终结 session”：

```kotlin
val isTerminalizing = uiState is RecognitionUiState.Saving ||
    uiState is RecognitionUiState.Discarding

PlateInputDialog(
    ...
    dismissEnabled = !isTerminalizing,
)
```

同时建议把按钮文案区分为：

- `Saving`：显示“正在保存…”；
- `Discarding`：显示“正在放弃…”或直接不展示确认对话框，改为全屏轻提示。

#### 验收标准

- 点击取消后，用户不能再点保存；
- 删除图片较慢时 UI 有明确 loading 状态；
- `Saving` / `Discarding` 的 UI 状态一致。

### P2-2：`setRecognized()` 返回值被忽略，状态迁移失败没有补偿

#### 现象

`RecognitionSessions.setRecognized()` 返回 `Boolean`：

- `app/src/main/java/com/example/platerecognizer/domain/Contracts.kt:71`

真实实现也确实通过 expected-state SQL 返回成功或失败：

- `app/src/main/java/com/example/platerecognizer/data/RecognitionSessionRepository.kt:48-61`

但 `PlatesViewModel.processRecognition()` 调用后没有检查结果：

- `app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt:245-249`

#### 影响

正常单进程 UI 下触发概率不高，因为 `recognitionMutex` 已经串行化识别任务。但在下面场景中仍可能出问题：

- 进程恢复逻辑与识别协程交错；
- DAO 层迁移/事务异常导致状态不匹配；
- 未来引入后台识别或多窗口后出现并发路径。

如果 `setRecognized()` 返回 `false`，当前 ViewModel 不提示错误、不标记失败、不清理图片，UI 可能停留在 `Recognizing` 或被旧状态覆盖。

#### 建议

检查返回值并补偿：

```kotlin
val updated = sessions.setRecognized(session.id, candidate, quality, err)
if (!updated) {
    runCatching { sessions.markFailed(session.id, "识别结果写入失败，请重试") }
    emit(UiEvent.Toast("识别结果写入失败，请重试"))
}
```

#### 验收标准

- fake session 让 `setRecognized()` 返回 `false` 时，ViewModel 能进入 `Failed` 或 `Ready`；
- 用户能看到失败提示；
- 不会永久停在 `Recognizing`。

### P2-4：domain 接口仍泄漏 Android 与 data 层类型

#### 现象

`domain/Contracts.kt` 已经做了抽象层，但接口仍直接依赖：

- `android.net.Uri`
- `data.PlateRecord`
- `data.ActiveSession`
- `data.SessionState`

位置：

- `app/src/main/java/com/example/platerecognizer/domain/Contracts.kt:3-6`
- `app/src/main/java/com/example/platerecognizer/domain/Contracts.kt:18-83`

当前通过 String 重载缓解了 JVM 测试里 `Uri` 的问题：

- `app/src/main/java/com/example/platerecognizer/domain/Contracts.kt:44-58`

但 domain 层仍不是纯 Kotlin。

#### 影响

- 单元测试仍需要绕过 Android stub；
- 未来拆模块时，domain 层不能独立为 JVM module；
- data 实体变化会直接波及 ViewModel 和 domain contract。

#### 建议

逐步引入 domain model：

```kotlin
data class PlateRecordModel(...)
data class RecognitionSessionModel(...)
enum class RecognitionSessionState(...)
@JvmInline value class ImageRef(val value: String)
```

然后：

- Repository 实现层负责 Entity ↔ Model 映射；
- domain 接口只暴露 Model；
- Android `Uri` 只留在 `ImageStore` Android 实现里。

#### 验收标准

- `domain` 包不 import `android.*`；
- `domain` 包不 import `data.*`；
- ViewModel 测试无需 `testOptions.unitTests.isReturnDefaultValues = true` 也能运行。

### P2-5：Release workflow 仍缺少 `versionCode` 单调递增校验

#### 现象

v0.2.9 Release workflow 已校验：

- 签名证书 SHA-256；
- APK `versionName` 与 tag 一致。

位置：

- `.github/workflows/release.yml:70-112`

但没有校验 `versionCode` 是否递增。当前 `app/build.gradle.kts` 中：

- `versionCode = 29`
- `versionName = "0.2.9-debug"`

位置：

- `app/build.gradle.kts:20-21`

#### 影响

如果以后 tag 与 `versionName` 匹配，但忘记递增 `versionCode`，Android 覆盖安装可能失败，用户拿到新 APK 却无法升级。

#### 建议

1. 用 `aapt2 dump badging` 读取 APK 的 `versionCode`；

3. 若当前 `versionCode <= previousVersionCode`，直接失败。

如果暂时不想下载旧 APK，也可以维护一个简单规则：

- tag `v0.2.9-debug` 要求 `versionCode >= 29`；
- 或要求 `versionCode` 与 patch 号一致。

#### 验收标准

- 忘记增加 `versionCode` 时，Release CI 失败；
- Release notes 中展示 `versionName` 与 `versionCode`。

### P2-7：CameraX ROI 裁剪链路缺少设备级验证

#### 现象

`CameraPreviewCard` 通过 `UseCaseGroup` 绑定 `Preview` 与 `ImageCapture`，并尝试使用 `PreviewView.viewPort`：

- `app/src/main/java/com/example/platerecognizer/ui/CameraPreviewCard.kt:86-92`

`PhotoCapturer` 假定 PreviewView 与 ImageCapture 输出按相同 ViewPort 对齐：

- `app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt:42-47`

如果 `viewPort` 为 null，当前代码会继续绑定，但不设置 viewport：

- `app/src/main/java/com/example/platerecognizer/ui/CameraPreviewCard.kt:87-91`

- `app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt:142-152`

#### 影响

在某些设备、方向、比例或 CameraX 行为差异下，取景框里看到的车牌与实际裁剪区域可能不完全一致，导致：

- 用户明明把车牌放进框内，OCR 却裁到偏移区域；

- bug 难以通过 JVM 测试发现。

#### 建议

1. 增加 debug 开关：保存原图与 ROI 图，便于人工对比。
2. 如果 `previewView.viewPort == null`，不要悄悄降级；至少记录错误或提示用户重新进入相机。
3. 增加设备验证清单：

- 16:9 手机；
- 20:9 手机；
- 小屏设备；
- 横竖屏策略调整后；
- 前后台切换后；
- 旋转 EXIF 为 90/270 的照片。

4. 对 `PhotoCapturer.cropToFile()` 的纯几何部分抽出函数做 JVM 测试。

#### 验收标准

- debug 模式可同时保存原图和 ROI 图；
- 至少 3 种屏幕比例下 ROI 与预览框对齐；
- `viewPort == null` 有明确 fallback 策略。

### P2-9：没有 Compose UI / 真实设备权限流程测试

#### 现象

### P3-2：CSV 导出仍有边界可加固

#### 现象

Android 10+ 使用 MediaStore + `IS_PENDING` 是正确方向：

- `app/src/main/java/com/example/platerecognizer/data/CsvExporter.kt`

但仍有两个边界可以加强：

1. `resolver.update(uri, values, null, null)` 的返回值没有检查；
2. Android 9 及以下直接 `writeText()` 到 Downloads，失败时可能留下半截文件。

#### 建议

- 检查 `resolver.update()` 返回值，为 0 时抛出异常；
- pre-Q 分支采用 `.tmp` 写入后 rename；
- 文件名加入毫秒或短 UUID，避免同一秒内连续导出覆盖或冲突。

## 7. 完成定义

下一版建议至少满足：

- `FAILED` 状态可见、可清理、错误原因可追踪；
- `markFailed(error)` 真正写入数据库；
- `setRecognized()` 失败有补偿；
- Room migration 与 DAO CAS 有 instrumentation 测试；
- Release workflow 校验 `versionCode`；
- Lint warning 明显减少或逐项解释；
- 在具备 JDK 17 的环境中可重新跑通：

```bash

```

整体判断：v0.2.9 已经从“补功能”进入“补可靠性”的阶段。现在最值得优先处理的不是继续堆 UI，而是把失败状态、数据库迁移、发布版本约束这三条链路闭环。这样后续每次 release 才会更像一个可长期维护的 Android 项目，而不是一只靠运气跑得挺快的小车。
