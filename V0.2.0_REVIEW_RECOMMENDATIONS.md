# Plate Recognizer Android v0.2.0 修改建议

## 1. 审阅结论

审阅对象：GitHub Release `v0.2.0-debug`，对应提交 `62862fa68d39b76ceebb026aaca0ffcce64883c0`。

本版已经解决上一版最严重的发布追溯问题：Release Tag、远端分支和本地源码指向同一提交，工作区干净，Gradle Wrapper 已提交。本地 APK 与 GitHub Release 的 SHA256 完全一致：

```text
0308415bc649de296fa0a919da3bbef7110409678ecd630348e227e8db70ebcb
```

现有验证结果：

- `PlateValidatorTest`：16/16 通过；
- `PlateRecognizerCandidateTest`：10/10 通过；
- Android Lint：0 errors、67 warnings；
- `assembleDebug`：成功；
- APK 已作为 prerelease/debug 构建发布。

不过，相机裁剪链路存在一个确定性阻断缺陷：裁剪成功后返回的 URI 仍指向已经删除的原图。建议先修复该问题，再发布 `v0.2.1-debug`。

## 2. 优先级总览

| 优先级 | 修改项 | 影响 |
|---|---|---|
| P0 | 裁剪后返回正确的文件 URI | 当前抓拍 OCR 可能固定读取已删除文件 |
| P1 | 将图片解码、旋转和压缩移出主线程 | 防止卡顿、ANR 和内存峰值过高 |
| P1 | 使用 CameraX ViewPort 对齐预览与照片 | 取景框显示区域与实际裁剪区域可能不一致 |
| P1 | 将待确认结果改为 ViewModel 状态 | 旋转、关闭、保存失败或下一次识别会丢数据 |
| P1 | 合并当前分支到默认 `main` | 默认克隆仍得到旧初始版本 |
| P2 | 重新评估 OCR TextBlock 几何过滤 | 文字框比例不等于车牌外框比例 |
| P2 | 收紧普通号牌序号规则 | 当前允许五个字母等与文档不一致的组合 |
| P2 | 完善相机 Provider 异常处理 | 初始化失败仍可能主线程崩溃 |
| P2 | 完善临时图片与失败清理 | 取消、复制失败和放弃确认会留下孤立文件 |
| P2 | 增加相机/图片集成测试及 CI | 当前单元测试无法覆盖本轮核心缺陷 |

## 3. 详细修改建议

### 3.1 P0：修复裁剪后返回已删除原图 URI

涉及文件：[`PhotoCapturer.kt`](app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt)

#### 问题原因

当前流程是：

1. CameraX 把照片保存到 `rawFile`；
2. `cropToFile()` 创建新的 `*_roi.jpg`；
3. `cropToFile()` 删除 `rawFile`；
4. `onImageSaved()` 优先返回 `result.savedUri`。

对于使用 `OutputFileOptions.Builder(File)` 的 CameraX 文件输出，`result.savedUri` 指向最初的 `rawFile`。因此裁剪成功后，调用方拿到的是已删除文件的 URI，随后 `InputImage.fromFilePath()` 会读取失败。

#### 最小修复

不要再优先使用 CameraX 返回的原始 URI：

```kotlin
override fun onImageSaved(result: ImageCapture.OutputFileResults) {
    if (!completed.compareAndSet(false, true)) return
    val finalFile = if (roi != null) {
        runCatching { cropToFile(rawFile, roi) }
            .getOrDefault(rawFile)
    } else {
        rawFile
    }
    cont.resume(Uri.fromFile(finalFile))
}
```

由于输出目标本来就是应用私有 `File`，这里不需要依赖 `result.savedUri`。

#### 验收标准

- 裁剪成功后返回 URI 的路径以 `_roi.jpg` 结尾；
- URI 对应文件存在，原始文件不存在；
- 抓拍后 ML Kit 可以成功打开该 URI；
- 裁剪失败回退时返回仍然存在的原始文件；
- 增加测试验证“裁剪文件替换后返回最终路径”。

### 3.2 P1：不要在主线程处理整张照片

涉及文件：[`PhotoCapturer.kt`](app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt)

#### 当前问题

CameraX 回调使用 `ContextCompat.getMainExecutor(context)`，而回调中执行了：

- `source.readBytes()`；
- 完整 JPEG 解码；
- EXIF 旋转；
- 创建裁剪 Bitmap；
- JPEG 重新压缩；
- 文件删除。

高像素照片会在主线程制造明显卡顿；原图字节、原 Bitmap、旋转 Bitmap 和裁剪 Bitmap 还可能同时存在，造成很高的内存峰值。

#### 建议重构

把“CameraX 保存原图”和“后台裁剪”拆成两个 suspend 阶段：

```kotlin
suspend fun takePicture(roi: PreviewRoi?): Uri {
    val rawFile = captureRawFile()
    val finalFile = if (roi == null) {
        rawFile
    } else {
        withContext(Dispatchers.IO) {
            cropToFile(rawFile, roi)
        }
    }
    return Uri.fromFile(finalFile)
}
```

进一步建议：

- 使用 `BitmapFactory.Options.inJustDecodeBounds` 先读取尺寸；
- 按 OCR 所需分辨率设置 `inSampleSize`，无需解码完整 12MP/48MP 图片；
- 避免 `readBytes()`，直接用 `BitmapFactory.decodeFile()`；
- CPU 密集的旋转和压缩可放在 `Dispatchers.Default` 或独立 Executor；
- 检查 `Bitmap.compress()` 返回值，失败时不要删除原图；
- 使用 `try/finally` 保证 Bitmap 正确回收或交给 GC。

#### 验收标准

- 抓拍裁剪时主线程没有超过 16ms 的长任务；
- 12MP 和 48MP 照片不会触发 OOM；
- 压缩失败时原图仍保留并可用于 OCR；
- 屏幕旋转或离开页面取消任务时不崩溃。

### 3.3 P1：通过 CameraX ViewPort 实现取景框所见即所得

涉及文件：

- [`MainScreen.kt`](app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt)
- [`PhotoCapturer.kt`](app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt)

#### 当前问题

PreviewView 默认使用 `FILL_CENTER`，会根据预览 Buffer、设备旋转和 View 尺寸进行缩放及中心裁切。ImageCapture 的输出分辨率和视野不一定与 Preview 相同。直接把 PreviewView 的归一化 ROI 乘以照片尺寸，无法保证裁剪到用户看到的取景框。

这不仅影响横屏；在竖屏手机中，把旋转后的 3:4 相机画面填入 4:3 UI 容器时也可能产生显著裁切偏差。

#### 建议修改

在 PreviewView 完成布局后取得 `viewPort`，将 Preview 和 ImageCapture 放入同一个 `UseCaseGroup`：

```kotlin
val viewPort = previewView.viewPort ?: return@addListener
val group = UseCaseGroup.Builder()
    .addUseCase(preview)
    .addUseCase(imageCapture)
    .setViewPort(viewPort)
    .build()

provider.bindToLifecycle(lifecycleOwner, selector, group)
```

CameraX 将首先保证 Preview 和 ImageCapture 对应相同的传感器区域；之后再把内部取景框的归一化坐标应用到照片上。

同时建议：

- 明确设置 `PreviewView.ScaleType.FILL_CENTER`；
- 监听显示旋转并更新 ImageCapture target rotation；
- 如果当前只支持竖屏，应暂时在 Manifest 锁定竖屏，而不是只在 Release Note 中声明限制；
- 在 4:3、16:9、不同传感器方向的设备上验证裁剪中心。

参考：[CameraX ViewPort 与 UseCaseGroup](https://developer.android.com/media/camera/camerax/configuration#crop_rect)

#### 验收标准

- 在裁剪结果上绘制 ROI 调试边框时，与屏幕取景框中的目标一致；
- 竖屏、横屏及至少两种不同宽高比设备均通过；
- 车牌放在取景框边缘时不会被意外裁掉；
- 前后摄像头或不同传感器方向不会导致镜像/旋转错位。

### 3.4 P1：把待确认识别结果建模为持久 UI 状态

涉及文件：

- [`PlatesViewModel.kt`](app/src/main/java/com/example/platerecognizer/ui/PlatesViewModel.kt)
- [`MainScreen.kt`](app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt)
- [`ImageStore.kt`](app/src/main/java/com/example/platerecognizer/data/ImageStore.kt)

#### 当前问题

`RequestPlateInput` 仍作为一次性 Channel 事件发送，UI 收到后存入本地 `remember`。以下操作会丢失待确认状态或关联图片：

- Activity 重建或屏幕旋转；
- 用户关闭确认框；
- 用户确认后数据库保存失败；
- 第一次识别已进入确认框，但用户又启动第二次识别；
- 进程在确认前被回收。

Mutex 只覆盖到识别结果发出为止，不覆盖人工确认阶段。

#### 建议修改

在 ViewModel 中加入状态：

```kotlin
data class PendingRecognition(
    val initial: String,
    val confidence: Float,
    val imageUri: Uri,
    val error: String?,
)

private val _pending = MutableStateFlow<PendingRecognition?>(null)
val pending: StateFlow<PendingRecognition?> = _pending.asStateFlow()
```

配套方法：

```kotlin
fun confirmPending(plateNo: String, note: String?)
fun discardPending()
```

行为要求：

- 数据库保存成功后才清空 `_pending`；
- 保存失败时保留对话框，允许重试；
- `discardPending()` 删除自有图片后再清空状态；
- 存在 pending 时禁用新拍摄/导入，或先弹出“放弃当前结果”确认；
- Toast 仍可使用 Channel，但对话框请求不应是一次性事件。

#### 验收标准

- 旋转屏幕后确认框及识别内容仍存在；
- 保存失败后用户可以直接重试；
- 关闭确认框会删除对应图片；
- pending 存在时不会被下一次识别覆盖；
- 自动保存和人工保存只会产生一条数据库记录。

### 3.6 P2：重新评估 TextBlock 几何过滤

涉及文件：[`PlateRecognizer.kt`](app/src/main/java/com/example/platerecognizer/ocr/PlateRecognizer.kt)

#### 当前问题

代码用车牌物理外框比例 `2.6–3.8` 检查 ML Kit `TextBlock.boundingBox`。但 ML Kit 的 boundingBox 包围的是识别文字，不是车牌边缘。文字所占宽高、边距、字体、透视角度和分块方式都会改变该比例。

因此可能出现：

- 实际车牌已经正确位于 ROI，却被几何过滤拒绝；
- 所有正常结果都进入全文 fallback，被统一压到 `0.7`；
- 其他长条文字仍可能恰好落入比例范围；
- Release Note 所述的广告牌/车架号过滤效果缺乏真实图片测试证明。

#### 建议修改

短期方案：既然相机输入已经裁剪到明确 ROI，可以先依赖 ROI + 严格字符校验，取消 TextBlock 外框比例硬过滤，或者将它只作为排序分而非淘汰条件。

中期方案：

- 收集真实车辆图片；
- 记录合法车牌 TextBlock 的 aspect、areaRatio、rotation；
- 根据数据分布确定阈值；
- 使用 `Line`/`Element` 的位置关系，而不是假定整个 TextBlock 等于车牌；
- 最终接入车牌检测模型时，再根据检测框做几何过滤。

#### 验收标准

- 至少使用白天、夜间、倾斜、远距离、普通和新能源车牌组成的测试集；
- 分别统计召回率、误报率和人工确认率；
- 几何过滤方法有独立测试，不只测试纯字符串函数；
- 合法车牌不会因为文字框比例与外框比例不同而被系统性拒绝。

### 3.7 P2：让普通号牌规则、测试和文档保持一致

涉及文件：

- [`PlateValidator.kt`](app/src/main/java/com/example/platerecognizer/util/PlateValidator.kt)
- [`PlateValidatorTest.kt`](app/src/test/java/com/example/platerecognizer/PlateValidatorTest.kt)

#### 当前问题

注释写明普通号牌序号为全数字、1 位字母或 2 位字母混合，但实现只检查字符是否属于合法字符集，因此 `京AAAAA` 等五字母序号也会通过。

现有测试中的 `沪AABC23` 实际包含三位序号字母，却被归入“1 或 2 位字母”测试，说明测试自身也没有约束目标规则。

#### 建议修改

如果产品范围确实要求普通号牌最多两位序号字母，可先增加基础约束：

```kotlin
val letterCount = serial.count { it.isLetter() }
if (letterCount > 2) {
    return "普通号牌序号最多包含 2 位字母"
}
```

然后根据 GA 36-2018 表 4 明确是否还需要限制字母出现位置和启用顺序。不要让注释声称“严格按标准”，但实现仅做宽松字符集检查。

补充测试：

- 全数字；
- 一位字母的所有允许位置；
- 两位字母的允许组合；
- 三位及以上字母；
- `I/O`；
- 非法发牌机关代码；
- 普通牌与新能源牌边界。

#### 验收标准

- `京AAAAA`、`沪AABC23` 等超出目标规则的号码被拒绝；
- 测试名称、注释、实现和 Release Note 描述一致；
- 如果业务决定允许更宽松格式，文档应明确说明其不是严格标准校验。

### 3.8 P2：覆盖 CameraProvider 初始化异常

涉及文件：[`MainScreen.kt`](app/src/main/java/com/example/platerecognizer/ui/MainScreen.kt)

#### 当前问题

`cameraProviderFuture.get()` 位于 `try` 之外。如果 Provider Future 失败，异常会在主线程 Listener 中直接抛出。

#### 建议修改

把 Provider 获取、Preview 创建和绑定放进同一个异常边界：

```kotlin
cameraProviderFuture.addListener({
    try {
        val provider = cameraProviderFuture.get()
        // 创建 Preview、UseCaseGroup 并绑定
    } catch (e: Exception) {
        vm.onCameraUnavailable(e)
    }
}, ContextCompat.getMainExecutor(ctx))
```

UI 应显示“相机不可用”，同时仍允许相册导入、记录浏览和 CSV 导出。

#### 验收标准

- 无摄像头模拟器不崩溃；
- CameraProvider Future 失败时有明确提示；
- 相机不可用时非相机功能仍可访问。

### 3.9 P2：完善孤立图片清理

涉及文件：

- [`ImageStore.kt`](app/src/main/java/com/example/platerecognizer/data/ImageStore.kt)
- [`PhotoCapturer.kt`](app/src/main/java/com/example/platerecognizer/camera/PhotoCapturer.kt)
- [`PlateRepository.kt`](app/src/main/java/com/example/platerecognizer/data/PlateRepository.kt)

#### 建议修改

- 相册导入先写 `.tmp`，完整复制成功后再重命名；
- 复制、压缩或协程取消时删除临时文件；
- CameraX 在协程取消后仍保存成功时，回调应删除生成文件；
- 确认对话框放弃时删除 pending 图片；
- 应用启动时，对比数据库 URI 与 `filesDir/plates`，清理超过安全期限的无引用文件；
- 路径校验使用规范化 `Path.startsWith(rootPath)`，而不是字符串前缀比较。

#### 验收标准

- 失败、取消和放弃确认后目录文件数量不增加；
- 删除数据库记录后对应自有图片消失；
- 相册原图和应用私有目录之外的文件绝不会被删除。

### 3.10 P2：添加 CI 与核心集成测试

当前 26 个 JVM 测试全部通过，但没有覆盖以下新增核心逻辑：

- CameraX 文件输出 URI 与裁剪结果；
- EXIF 90/180/270 度旋转；
- PreviewView ROI 到 ImageCapture 的坐标转换；
- 大图片解码和内存峰值；
- ML Kit TextBlock 几何过滤；
- pending 状态在 Activity 重建后的恢复；
- 放弃确认和保存失败后的文件清理。

建议 GitHub Actions 至少执行：

```bash
./gradlew --no-daemon \
  :app:testDebugUnitTest \
  :app:lintDebug \
  :app:assembleDebug
```

Release APK 应由 Tag 触发的 CI 从干净检出环境构建并上传，而不是人工上传本地工作区产物。建议同时上传：

- APK；
- SHA256 文件；
- 测试结果；
- Lint 报告；
- 构建使用的 commit SHA。

#### 验收标准

- PR 未通过测试或 Lint 时不能合并；
- Tag 构建出的 APK 自动关联精确 commit；
- Release 构建不依赖开发者本地文件；
- 关键相机裁剪路径至少有 Instrumentation 或 Robolectric 测试。
